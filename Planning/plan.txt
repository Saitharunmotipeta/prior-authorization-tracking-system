This is where many projects fail.

People build:

```text
Controller
↓
Controller
↓
Controller
```

and then later realize:

```text
No architecture
No audit
No resilience
No analytics
```

You already did the hard part (design). Now follow a strict build sequence.

---

# Phase 1 — Foundation

Build once.

Never touch again.

## Step 1

Create Solution

```text
PriorAuthorizationSystem.sln
```

Create Projects

```text
Specialist.API
Payer.API
Manager.API

PriorAuthorization.Data
PriorAuthorization.Shared
```

---

## Step 2

Create Shared Enums

```text
ConditionType

RequestStatus

Priority

ReminderCategory

ReminderStatus

AuditActionType

UserRole
```

Because entities depend on them.

---

## Step 3

Create Entities

```text

Department

Patient

Payer

Policy

Encounter

AuthorizationRequest

AuthorizationService

Reminder

AuditHistory
```

Don't create repositories yet.

---

## Step 4

Create Configurations

```text
FacilityConfiguration

PatientConfiguration

...
```

Define:

```text
PKs

FKs

Indexes

Constraints

Datatypes
```

---

## Step 5

Create DbContext

```text
ApplicationDbContext
```

Register all entities.

---

## Step 6

Migration

```bash
Add-Migration InitialCreate
Update-Database
```

Verify tables.

# Phase 5 — Resilience

Now.

Not before.

---

# Where Resilience Applies

## Database Calls

Repository:

```text
Get Encounter
```

Database timeout?

Retry.

---

Example:

```text
Retry
3 Attempts
```

---

## Analytics

Heavy query.

```text
Dashboard Query
```

Timeout after:

```text
5 Seconds
```

Don't hang forever.

---

## Email Reminder

SMTP Down?

Fallback.

---

Example:

```text
Email Failed
↓
Create Reminder Record
↓
Log Audit
```

User still sees success.

---

# Where NOT To Use Resilience

Don't put Polly everywhere.

Not:

```text
Every Controller

Every Method

Every Query
```

Only around:

```text
Database

Email

External Calls
```

Build first.

Because everything starts here.

Order:

```text

```

This validates:

```text
Patient

Policy

Encounter

Request
```

flow.

# Final Build Order (Freeze This)

```text
1. Solution Structure

2. Enums

3. Entities

4. Configurations

5. DbContext

6. Migration

7. Seed Data

8. Repositories

9. Services

10. Resilience

11. Specialist API

12. Payer API

13. Audit Logic

14. Manager API

15. Readiness Engine

16. Risk Engine

17. Testing

18. Documentation
```

This order minimizes rework because every layer depends only on layers that are already stable. Build it in this sequence and you'll rarely need to go back and redesign earlier pieces.
